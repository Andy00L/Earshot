## 1. Root cause

The asset slicer's chroma key function (`scripts/chroma.py:chroma_key_hsv`)
operates exclusively on green backgrounds. It thresholds pixels in HSV
color space using H:35-85, which covers the green hue band. The two
whisper charm source PNGs (`raw/items/whisper_charm.png` and
`raw/props/whisper_charm_explainer.png`) were generated by ChatGPT with
a magenta background. Magenta sits at HSV H:~150 in OpenCV's 0-180
scale, well outside the green detection range. The slicer passed
magenta pixels through as fully opaque foreground, producing sliced
PNGs with the magenta background intact.

The three other puzzle-props sprites (trapdoor_sealed, broken_tape,
tape_recorder) sliced cleanly because their source PNGs have green
backgrounds that fall within the HSV green range.

Sampled source pixel values confirmed the diagnosis:

| Sprite | Corner pixel (R, G, B) | Expected chroma |
|--------|------------------------|-----------------|
| whisper_charm | (237, 7, 238) | magenta |
| whisper_charm_explainer | (239, 2, 244) | magenta |

Before fix, sliced output corner pixels: (238, 5, 239, alpha=255).
Magenta preserved with full opacity.

Possible causes investigated (Phase 0):

- Tolerance too low for imprecise magenta: N/A (HSV-based, not tolerance-based)
- Source PNG has imperfect magenta: yes, but within normal ChatGPT variance
- Slicer config has wrong source path: no
- Wrong chroma_key tuple in slicer config: N/A (config uses boolean, not tuple)
- Slicer single_chroma mode broken: yes (green-only HSV detection)
- Atlas not regenerated / cache issue: no

Identified path: E (slicer single_chroma mode lacks magenta support)

## 2. Fix applied

Added a `chroma_key_magenta` function to `scripts/chroma.py` that
mirrors the existing `chroma_key_hsv` pipeline but uses magenta HSV
bounds (H:125-170, S:80-255, V:80-255). Includes a magenta-specific
despill step that reduces R and B channels where min(R,B) exceeds G
by more than the threshold, clamping down fringe pixels that blend
into the foreground edge.

Added a `chroma_color` field to `scripts/atlas_config.py` for both
whisper charm entries. Defaults to `"green"` when absent, set to
`"magenta"` for these two sprites.

Updated `scripts/slice.py:_handle_single_chroma` to check
`profile.get("chroma_color", "green")` and route to
`chroma_key_magenta` when the value is `"magenta"`.

No TypeScript changes. The bug was entirely in the Python slicer
pipeline. The HUD code, popup code, and game logic from Hotfix U.20
are correct and unmodified.

## 6. Trade-offs

The magenta HSV range H:125-170 is set wider than strictly needed to
accommodate ChatGPT's imprecise magenta generation. The sampled pixels
had H ~149-150, so the range covers variance in future sprites as well.
No content pixels in either sprite fall within this band.

The magenta despill reduces R and B channels on edge pixels. For bone
ivory and brown tones (R > G > B), the despill condition
min(R,B) > G + 5 is rarely triggered because B is typically below G
in these warm tones. The despill primarily affects the thin
anti-aliased fringe where magenta background blends into the subject
edge.

The `chroma_color` config field defaults to `"green"` when absent,
preserving backward compatibility for all existing entries. Only the
two whisper charm entries specify `"magenta"`. Future magenta-background
sprites can reuse this field.
