### 3.4 Descent bug analysis

The emerging interpolation runs from `grateBottomY` (815.72) to `floorY` (865.72). It does NOT stop short. The end Y is exactly `floorY`. There is no offset bug in the Y interpolation itself.

However, there is an important subtlety. On `enterState("emerging")` (line 257-258):
```typescript
case "emerging":
  this.playAudio("jumper_vent_creak");
  this.container.y = this.grateBottomY;   // <-- sets Y to 815.72
  this.container.alpha = 1;
  break;
```

The container.y is set to grateBottomY at state entry, and then `updateEmerging()` interpolates from grateBottomY to floorY. At t=1.0, the container reaches floorY (865.72). The math is correct.

**Then, immediately after emerging finishes, the state transitions to getting_up (line 412-413), which sets container.y = floorY (line 418). This is also correct.**

The Y interpolation appears arithmetically sound. If the creature does not visually reach the floor, the problem is not in the Y position but in the **sprite height clamp interaction**. See Phase 7 for the full render-time analysis.

### 3.5 Emerging vs retreating symmetry

Emerging: 815.72 to 865.72, 800ms, linear.
Retreating: 865.72 to 815.72, 1500ms, linear.

They travel the same 50px distance but at different speeds. Retreating is 1.875x slower. Both are linear. The asymmetry is in duration only, not in start/end positions or easing. Duration asymmetry alone does not explain a "creature does not reach floor" visual.

### 3.6 Easing function

All Y interpolations use `t = min(1, stateTimer / duration)` with a direct linear lerp. There is no ease-in, ease-out, or other nonlinear easing anywhere in the jumper code. Both emerging and retreating use the same linear function.

## 8. Verdict

### Bug 1: Vent sprite size mismatch (getup frames scaled too wide)

**Root cause:** The `applyLockedSpriteSize()` method at `src/jumper.ts:518-529` applies uniform scaling (`this.sprite.scale.set(scale)`) using a single value derived from the height clamp. When the target height (220) is much larger than the native frame height (getup1 = 118), the uniform scale of 1.86x is applied to width as well, inflating the sprite to 607.7px wide. The grate (283.6px wide) cannot cover this.

**Symbols to change:**
- `src/jumper.ts:528` (`this.sprite.scale.set(scale)`)

**Target fix:** Instead of uniform `scale.set(scale)`, compute width and height scales independently. Lock the height to the clamp value and cap the width to a maximum that prevents overflow beyond the grate width (~283.6px). One approach: set `scale.y = targetHeight / textureHeight` and `scale.x` to the same value but capped so that `textureWidth * scale.x` does not exceed a per-state max width. Alternatively, set width and height directly (`this.sprite.height = targetHeight; this.sprite.width = Math.min(nativeWidth, maxWidth)`) and let Pixi compute non-uniform scale.

**Risk:** Changing to non-uniform scale will distort the creature's proportions. The getup frames are naturally wide and short (a crouching/prone creature), so squashing them horizontally may look unnatural. An alternative is to use a smaller height clamp for getup-based states so the scale factor stays closer to 1.0, but this shrinks the creature. Testing both approaches visually is necessary.

### Bug 2: Creature behind grate during fake_attacking

**Root cause:** The `updateZIndexForVentStacking()` method at `src/jumper.ts:458-463` includes `"fake_attacking"` in the `peekingStates` array, giving it zIndex 70. The dripSprite has zIndex 75. So the creature renders behind the grate during the entire fake_attacking sequence, even as it emerges to floor level.

**Symbol to change:**
- `src/jumper.ts:461` (the `peekingStates` array)

**Target fix:** Remove `"fake_attacking"` from the peekingStates array:
```typescript
const peekingStates: JumperState[] = ["dormant", "peeking"];
```

This gives fake_attacking a zIndex of 85, placing it in front of the dripSprite (75).

**Risk:** Low. The fake_attacking state is specifically designed to show the creature emerging. Rendering it in front of the grate is the intended visual. No other system reads the jumper's zIndex for gameplay logic.

### Bug 3: Descent does not visually reach floor

**Root cause (revised):** The Y interpolation is arithmetically correct. The container.y reaches floorY at the end of emerging. The visual problem is that during the early frames of the emerging animation, the `getup1` frame (118px native) is scaled to 220px height, producing a 607.7px-wide sprite. This extreme width distortion, combined with the creature starting at grateBottomY (50px above floorY), creates a visual where the creature appears to balloon outward rather than descend smoothly. The player perceives this as the creature "not reaching the floor" because the visual focus is on the distortion rather than the vertical movement.

Additionally, the emerging state uses frames getup1 through getup6, played forward once. The animation runs at 120ms per frame. 6 frames at 120ms = 720ms of animation. The Y interpolation duration is 800ms (`FLOOR_EMERGING_DURATION_MS`). At 720ms, the animation holds on the last frame (getup6, 198px tall, scale 1.11x, width 391px). The final 80ms of the interp plays with a held frame, which is fine.

The retreating state uses walk frames (203-223px) at clamp 200, producing near-unity scale. No width distortion. This explains why retreating "looks fine" while emerging "looks wrong": the distortion is in the frame-to-clamp ratio, not in the Y interpolation.

**Symbol to change:** Same as Bug 1 (`src/jumper.ts:528`). The fix for Bug 1 also fixes Bug 3.

**Risk:** See Bug 1.

### Follow-on issues (out of scope)

1. **No explicit `y = floorY` in crawling state.** The `updateCrawling()` method (line 424-442) modifies `container.x` but never sets `container.y`. It relies on `getting_up` having set it to `floorY` in the previous state. If the state machine ever transitions directly to crawling (bypassing getting_up), the Y would be wrong. Currently this cannot happen (emerging always leads to getting_up which leads to crawling), but the implicit dependency is fragile.

2. **dripSprite not cleaned up on jumper destroy.** The `Jumper.destroy()` method (`src/jumper.ts:222-225`) removes and destroys the jumper container but does not touch the dripSprite, which lives in `this.world` and is tracked separately in `this.jumperDripSprites[]` in game.ts. The game.ts cleanup code (line 1275-1279) does handle this, so no leak occurs in practice. But if a Jumper is destroyed outside the game's cleanup path, the dripSprite remains.

3. **Ceiling jumpers also use zIndex 85 for all states (including dormant/peeking).** The `updateZIndexForVentStacking()` method returns early if `ventPosition !== "floor"`. Ceiling jumpers stay at whatever zIndex was set in the constructor (85 for non-locker, non-floor). This means ceiling jumpers in dormant state render at zIndex 85, above the foreground layer (80). If a cubicle divider sits at the same X as a ceiling jumper, the idle/peeking creature would render in front of the divider, which may be visually incorrect. The floor variant handles this with the 70/85 split, but the ceiling variant does not.
